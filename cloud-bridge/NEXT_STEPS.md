# 云转桥 (Cloud Bridge) MVP 冲刺执行计划 (至3月初)

## 1. 核心目标
将当前的 **代码骨架 (Skeleton)** 转化为 **可上线演示的 MVP 产品**，重点解决 Mock 数据的替换、核心服务的真实对接以及生产环境部署。

## 2. 后续具体行动步骤 (Action Steps)

### 第一阶段：核心服务真实化 (预计 4 天)
1.  **AI 引擎对接**：
    *   **行动**：申请 ChatGLM (智谱AI) API Key。
    *   **开发**：重构 `MatchingService`，替换字符串分割逻辑，实现对 HTTP API 的调用，并解析返回的 JSON 数据（提取技术关键词、所属领域）。
    *   **调优**：编写 Prompt 提示词，优化 AI 对科技成果摘要的理解准确度。
2.  **区块链节点搭建**：
    *   **行动**：在服务器/本地搭建 FISCO BCOS (2节点) 测试链。
    *   **开发**：引入 `fisco-bcos-java-sdk`，替换 `MockBlockchainService`。配置证书与私钥，实现真实的交易签名与上链。
    *   **合约**：编译 `Evidence.sol` 为 Java Wrapper 类，确保后端能直接调用合约方法。

### 第二阶段：数据填充与图谱构建 (预计 3 天)
1.  **数据清洗**：
    *   **行动**：整理申报书中提到的“生物医药”、“新材料”领域的演示数据（约 20-50 条高质量数据）。
    *   **脚本**：编写 Python 脚本或 SQL 脚本，将非结构化文本转化为结构化的 CSV/JSON。
2.  **图谱导入**：
    *   **行动**：使用 Neo4j Cypher 语句将清洗后的数据批量导入图数据库。
    *   **关联**：建立 `(Technology)-[:BELONGS_TO]->(Field)` 和 `(Achievement)-[:USES]->(Technology)` 的关系。

### 第三阶段：界面优化与集成测试 (预计 3 天)
1.  **前端美化**：
    *   **行动**：优化 Element Plus 组件样式，增加加载动画（Loading）、操作反馈（Toast）和错误处理。
    *   **展示**：增强“匹配结果”的展示效果，使用 ECharts 可视化展示“推荐理由”中的图谱关联路径。
2.  **全链路测试**：
    *   **行动**：模拟完整用户旅程：注册 -> 发布需求 -> AI 匹配 -> 存证上链 -> 核验。

### 第四阶段：云端部署 (预计 2 天)
1.  **服务器环境**：
    *   **行动**：购买/配置阿里云 ECS (CentOS 7.9, 4核8G)。
    *   **部署**：安装 Docker & Docker Compose，拉取镜像并启动。
2.  **域名与网络**：
    *   **行动**：配置 Nginx 反向代理，配置 SSL 证书 (HTTPS)，开放必要端口。

## 3. 技术实现方案

| 模块 | 当前状态 (Skeleton) | 目标状态 (MVP) | 技术栈/工具 |
| :--- | :--- | :--- | :--- |
| **AI 匹配** | 简单的字符串包含匹配 | **LLM 语义理解 + 图谱多跳查询** | ChatGLM-6B API (HTTP), Neo4j Cypher |
| **区块链** | 返回随机 UUID | **真实联盟链交易** | FISCO BCOS v3, Java SDK, Solidity |
| **数据库** | 仅含 2 条测试数据 | **覆盖 2 个垂域的演示数据** | MySQL 8.0, Neo4j 5.x |
| **部署** | 本地 localhost | **公网 IP / 域名访问** | Docker Compose, Nginx, Aliyun ECS |

## 4. 所需资源

1.  **计算资源**：
    *   阿里云 ECS 实例：推荐 **eccs.g6.xlarge (4 vCPU, 16 GiB)**，用于运行 Java 后端、Neo4j (内存大户) 和 FISCO BCOS 节点。
    *   存储：至少 50GB 高效云盘。
2.  **API 服务**：
    *   智谱 AI (ChatGLM) API Key (Open Platform)。
3.  **基础服务**：
    *   域名 (可选，建议配置以提升演示专业度)。
    *   SSL 证书 (Let's Encrypt 免费证书)。

## 5. 成功验收标准 (KPI)

1.  **功能完整性**：
    *   [ ] 能够通过 AI 准确识别出不少于 5 种不同的技术关键词。
    *   [ ] 区块链存证操作能够返回真实的 Transaction Hash，并在区块链浏览器中可查。
2.  **演示流畅度**：
    *   [ ] AI 匹配接口响应时间 < 3秒。
    *   [ ] 存证上链接口响应时间 < 2秒。
    *   [ ] 演示过程中无 Crash 或 500 错误。
3.  **数据质量**：
    *   [ ] 包含至少 20 条真实/模拟的高质量科技成果数据。
    *   [ ] 知识图谱中至少包含 3 层深度的实体关系。

## 6. 风险点与应急预案

| 风险点 | 可能性 | 影响程度 | 应急预案 |
| :--- | :--- | :--- | :--- |
| **FISCO BCOS 环境配置失败** | 中 | 高 | **降级方案**：回退使用 MockBlockchainService，但在前端展示“模拟网络”标识，重点演示存证流程而非底层技术。 |
| **AI API 响应过慢或不稳定** | 高 | 中 | **缓存机制**：对高频演示的 Query 结果进行 Redis 缓存；演示时优先使用已缓存的 Case。 |
| **服务器资源不足 (OOM)** | 中 | 高 | **资源隔离**：限制 Neo4j 和 Java 进程的堆内存；必要时开启 Swap 交换分区。 |
| **演示数据匹配效果差** | 中 | 中 | **人工干预**：在数据库中预埋几组“完美匹配”的数据对，确保演示时至少有成功的 Case。 |
