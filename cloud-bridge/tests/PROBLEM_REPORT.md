# 云转桥平台问题诊断报告 (Problem Diagnosis Report)

## 1. 摘要

本次测试从**安全性、健壮性、业务逻辑合理性**三个维度对系统进行了“对抗式”检查。
**结论**：系统存在**高危逻辑漏洞**和**严重设计缺陷**，虽然基本功能可用，但无法满足生产环境的安全与稳定性要求。

---

## 2. 严重问题 (CRITICAL)

### 2.1 缺乏基础数据校验 (Data Validation Missing)
- **现象**：系统允许发布价格为负数 (`-1000.0`) 的成果，允许发布无标题 (`title=null`)、无描述的“幽灵数据”。
- **风险**：
  - **业务逻辑崩溃**：负数价格会导致交易结算系统出现严重财务漏洞（如用户购买反而获利）。
  - **前端异常**：空数据可能导致前端页面渲染时出现 `NullPointerException` 或空白卡片。
  - **数据库污染**：大量垃圾数据无法被清理。
- **证据**：`vulnerability_test.py` 成功创建了负数价格和空标题的记录。
- **根因**：后端实体类 `Achievement.java` 未使用 `@NotNull`, `@Min(0)` 等 Bean Validation 注解，Controller 层也未进行手动校验。

### 2.2 内存爆炸风险 (Memory Bomb)
- **现象**：`MatchingService.java` 中的 `match` 方法调用了 `achievementRepository.findAll()`。
- **风险**：
  - **OOM (Out Of Memory)**：随着数据量增长（如达到 10 万条），该操作会将整张表加载到 JVM 内存中，直接导致服务崩溃。
  - **全表扫描**：数据库层面发生全表扫描，IO 飙升，阻塞其他请求。
- **根因**：未使用数据库层面的全文检索或条件查询，而是将计算压力全部转移到了应用层内存中。

---

## 3. 高风险问题 (HIGH)

### 3.1 中文分词逻辑缺陷 (Segmentation Fault)
- **现象**：输入无空格的中文长句（如“我想要寻找一个能够解决高并发问题的分布式数据库方案”）时，匹配结果为 0。
- **风险**：
  - **搜索不可用**：中文用户习惯输入整句，系统目前的 `split(" ")[0]` 逻辑会将整句作为一个关键词进行精确匹配，导致无法匹配到任何包含部分关键词（如“数据库”、“高并发”）的成果。
  - **用户流失**：核心搜索功能对中文支持极差。
- **根因**：`MatchingService.java` 在 AI 提取失败后的降级策略过于简单粗暴，未集成中文分词器（如 HanLP 或 Jieba）。

### 3.2 缺乏分页保护 (No Pagination)
- **现象**：`AchievementController` 的 `getAllPublished` 接口直接返回所有数据。
- **风险**：
  - **带宽耗尽**：如果数据库有 1GB 数据，请求一次就会传输 1GB，瞬间打满服务器带宽。
  - **客户端卡死**：前端浏览器接收大量数据解析 JSON 时会卡死。
- **根因**：未使用 `Pageable` 进行分页处理。

---

## 4. 中等风险问题 (MEDIUM)

### 4.1 存储型 XSS 风险 (Stored XSS)
- **现象**：系统允许将 `<script>alert('XSS')</script>` 存入数据库。
- **风险**：如果前端 Vue 代码使用了 `v-html` 或未正确转义，攻击者可注入恶意脚本窃取用户 Cookie 或 Token。
- **根因**：后端未对输入进行 HTML 标签清洗（Sanitization）。

### 4.2 浮点数精度丢失
- **现象**：价格字段使用 `Double` 类型。
- **风险**：在进行金额计算时（如累加、打折），浮点数运算会产生精度误差（如 `0.1 + 0.2 = 0.30000000000000004`），导致财务对账不平。
- **建议**：必须使用 `BigDecimal` 类型。

### 4.3 硬编码与异常吞噬
- **现象**：`MatchingService` 中 Neo4j 查询失败仅打印 `System.err`，业务继续运行。
- **风险**：生产环境日志难以监控，且知识图谱服务宕机时用户无感知，只会觉得“匹配不准”。

---

## 5. 改进建议 (Action Items)

1.  **立即修复校验**：在 `Achievement` 实体类添加 `@NotNull`, `@Min(0)`, `@Size(max=...)` 注解，并在 Controller 中使用 `@Valid`。
2.  **重构匹配逻辑**：废弃 `findAll()` 内存匹配，改用数据库全文索引（MySQL Full-Text Search 或 ElasticSearch）。
3.  **引入分页**：所有列表接口强制要求分页参数（`page`, `size`）。
4.  **优化中文支持**：集成 IK 分词器或 Jieba 分词器，对用户输入进行分词后再去数据库查询。
5.  **类型修正**：将 `Double price` 改为 `BigDecimal price`。
