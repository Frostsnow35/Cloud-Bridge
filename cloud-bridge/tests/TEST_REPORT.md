## 3. 详细测试数据

### 3.1 压力测试 (Stress Testing)

- **读取负载 (Read Load)**:
  - 场景: 模拟 20 个并发用户获取成果列表
  - 请求数: 100
  - 平均耗时: **63.11 ms**
  - P95 耗时: **89.36 ms**
  - 成功率: **100%**

- **写入负载 (Write Load)**:
  - 场景: 模拟 10 个并发用户发布需求
  - 请求数: 50
  - 平均耗时: **41.57 ms**
  - P95 耗时: **54.66 ms**
  - 成功率: **100%**

- **计算负载 (Compute Load)**:
  - 场景: 模拟 5 个并发用户进行 AI 智能匹配
  - 请求数: 20
  - 平均耗时: **1145.99 ms**
  - P95 耗时: **1298.70 ms**
  - 成功率: **100%**

### 3.2 边缘测试 (Edge Testing)

- **空描述输入**: 系统返回了所有 31 条成果数据。建议优化为返回提示信息或空列表。
- **超长文本输入 (10000 字符)**: 系统成功处理，耗时约 2.76s，返回了匹配结果。证明系统对大数据量输入有较好的鲁棒性。
- **特殊字符输入**: 系统正常处理，返回空列表，未发生 SQL 注入或解析错误。

### 3.3 匹配度测试 (Match Quality)

- **测试用例**: 输入 "Need software testing solution"
- **预期**: 匹配到包含 "Software Engineering" 的成果
- **实际**: 未返回结果 (WARN)。
- **分析**: 目前系统内大部分数据为生物医药类（如 PD-1, CAR-T, mRNA），虽然手动插入了一条软件工程数据，但可能由于 AI 模型对单一数据的权重计算或关键词提取策略，导致未进入 Top N 推荐列表。建议增加更多相关领域数据后再次测试。

## 4. 结论与建议

1.  **性能优异**: 系统在本地 Docker 环境下表现出极高的响应速度和吞吐量，能够轻松应对当前的并发压力。
2.  **稳定性强**: 在边缘情况（超长文本、特殊字符）和高并发下，系统均未崩溃，表现稳定。
3.  **功能完备**: 核心业务流程（发布、查询、匹配）均已打通。
4.  **改进建议**:
    - 优化匹配算法，提高冷门领域（数据量少）的匹配召回率。
    - 优化空输入的处理逻辑，避免返回全量数据造成带宽浪费。
